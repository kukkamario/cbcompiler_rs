| [FD-046](archive/FD-046_STRING_NUMBER_CONVERSION_PRIMITIVES.md) | Core-Runtime String↔Number Conversion Primitives | 2026-06-26 | Moves the conversions that **cross the `String` type** out of the interpreter's Rust and into the C++ core runtime as **bare exported symbols** (`cb_rt_int/long/float_to_string`, `cb_rt_string_to_long/float`) — so the interpreter and a future native/LLVM backend share **one** implementation and cannot silently diverge, most critically on **float→string formatting** (a `3.14` vs `3.140000` mismatch would have nothing to catch it; a native `.exe` can't call back into Rust). New Allegro-free TU `runtime/cb_convert.{cpp,h}`, **outside** the `CB_NO_ALLEGRO` guard so it ships in both the SDK-free and full builds and is headless gtest-able; declared in `cb-runtime-sys` as bare symbols like `cb_rt_string_test_refcount` — **NOT** `CB_FN` catalog rows, so they touch neither the catalog data, `CB_CATALOG_VERSION`, nor the FD-045 drift guard. **The boundary (confirmed with the user):** numeric↔numeric casts (incl. `Int(Float)` rounding) stay **backend-inline**; `Hex$`/`Bin$`/`Chr$`/`Asc` stay **ordinary catalog functions** (future FD); only the `Convert`/`ConvertExplicit`-opcode string conversions live here. **Float→String format (decoded empirically from the original CoolBasic over two probe runs):** 6 significant digits; **fixed-point iff** decimal exponent `E = floor(log10|x|) ∈ [-3,7]`, scientific otherwise; fixed strips trailing fractional zeros but always keeps ≥1 (`4.0`→`"4.0"`, never `"4"`); scientific is `m.e±EEE` (lowercase `e`, always-signed **3-digit** exponent, mantissa zeros stripped with `.` kept — `1e8`→`"1.e+008"`, `123456700.0`→`"1.23457e+008"`); decimal separator `.`, `-0.0`→`"0.0"`. Implemented via `snprintf("%.5e")`-then-reconstruct (6-sig-fig rounding **precedes** the fixed/sci branch, so `99999990.0`→`"1.e+008"`), locale- and exponent-width-independent — **not** Ryū/shortest-round-trip. **String→Float** changes from the interp's old **strict** full-parse to a lenient `strtod`-style **prefix** parse (`"22yo"`→`22.0`, `"1.5e2"`→`150.0`, junk→`0.0`) — matching real CB (the old `to_f64` was **wrong**). **String→Int** keeps the lenient leading-integer parse (stop at first non-digit incl. `.`, saturating), a **deliberate, documented divergence** from CB's exact-`.5`-rounds-up-positives quirk (`"22.5"`→`22`, not `23`). `value.rs` `as_cb_string`/`to_i64`/`to_f64` become thin wrappers over the shared symbols (still the single source of truth for the FFI marshaller + array-index coercion). **Verified (2026-06-26, Windows + Allegro SDK):** `cargo test --workspace` green with the interp running against the **full CMake-built** `cb_convert.cpp` → re-baselined interp/driver snapshots confirmed **byte-identical** to the C++ formatter (9 fixtures + 3 new interp tests for `Str(Float)`/`Float(s$)`/`Int(s$)`); native **`ctest -R Convert` 9/9** pinning the format spec; SDK-free build (`CB_RUNTIME_FORCE_SDK_FREE=1`) compiles the TU and passes; `clippy --workspace --all-targets -D warnings` exit 0 + `fmt --all --check` clean; catalog asserts unchanged, no `CB_CATALOG_VERSION` bump. **Prerequisite for / surfaced by** the LLVM-backend readiness `/fd-deep` (the sole *silent* interp/native divergence); pairs with the future register-type-inference pass (source-register type selects inline-cast vs. `call cb_rt_string_to_long`). Builds on [[FD-045]]'s metadata-vs-binding seam by deliberately staying **out** of the catalog/drift-guard machinery. |
